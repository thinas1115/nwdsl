このドキュメントは、nwdsl レンダラの (1) 物理図レイアウトの構造的欠陥の是正、(2) 通信経路(正常時/障害時・プロトコル込み)の可視化、の2テーマを次の担当が着手するための自己完結した指示書。

## 0. 依頼サマリ

`nwdsl` は NW 構成を YAML 1ソースで記述し、構成図(D2/Mermaid)と設計書向け表(Markdown)を生成するツール。基本機能は実装済みだが、**構成図が「意味的に正しい配置」を保証しておらず**、複雑な LAN で読めなくなる。加えて、設計書に不可欠な **通信経路(正常時/障害時・経路上のプロトコル)を表現・描画する機能が無い**。この2点を、DSL スキーマとレンダラの両面から設計し直して実装する。

**着手前に必ず**: 現状コードと下記の再現手順で問題を自分の目で確認し、D2/ELK のレイアウト制御能力を実機で検証してから、設計案(複数オプション+推奨)を提示して承認を得ること。いきなり実装しない。根本原因が未確定のまま症状にパッチを当てない。

## 2. 問題1: 物理図レイアウトの構造的欠陥

### 2.1 観測された症状(complex-lan の hq-physical / physical-all で確認済み)

1. **WAN の向きが定まらない**: WAN境界ルーターが拠点コンテナの一方の端、WANクラウド(IP-VPN網/インターネット)がコンテナ外の反対側の端に置かれ、**WAN回線の線が LAN 機器群を横断する**。単純構成では「LANスイッチがルーターより外側(WAN側)に見える」という形で顕在化した(発端の指摘)。
2. **サイトラベルと線の衝突**: 拠点コンテナのタイトル(例「本社」)の上を WAN 回線の線・ラベルが突き抜ける。
3. **全社図の破綻**: `physical-all` は全拠点の LAN 詳細を1枚に詰め込むため、拠点が縦積み+共有クラウドへ右方向に集約し、交差が多発。
4. LAN 内部の段組み(FW→コア→ディストリ→アクセス)自体は ELK layered が概ね綺麗に並べられている。**ただしこれはトポロジが WAN 境界を根とする DAG に近いための結果であって、制約で保証されたものではない**。トポロジが崩れれば保証はない。

### 2.2 根本原因

`render_d2.py` は `direction: right` を出すだけで **ELK にランク(層)決定を丸投げ**し、クラウドをコンテナ外の兄弟ノードとして置いている。レイアウトに「**WAN が端、LAN が内側へ降りる**」という階層・向きの概念が一切ない。したがって (a) WAN 側が一定方向に揃う保証がなく、(b) WAN境界機器がクラウドの隣に来ない。これは描画の小バグではなく、**レイアウトモデルの設計不在**。

## 3. 問題1の設計論点(決定軸・制約・未検証事項)

安易に「role から tier を振って階層強制」に飛びつかないこと。以下の軸を実測で詰めた上で設計する。

### 軸A: 階層をどこまで強制するか

- 2.1-4 の通り **LAN 内部の段組みは ELK が実際うまくやれている**。全機器に tier を強制するのは over-engineering の恐れ。
- 発端の不満は「WAN の向きが定まらない」こと。**最小介入 = WAN境界(クラウド接続を持つ機器)を常に一定端に固定** で足りるか、それとも段の明示管理まで要るかを判断する。

### 軸B: tier(層)の導出方法(段を管理する場合)

- **role→tier の単純マッピングは破綻する**: `l3switch` がコアにもディストリにも使われる(complex-lan の core01/02 と dist01/02 は同じ role)。role だけで段を決めるとコアとディストリが潰れ、今より悪化する。
- 候補: (a) DSL に明示 `tier:` を持たせ role からデフォルト提案、(b) **WAN境界からの物理ホップ距離(BFS深さ)で層を自動算出**(core=1, dist=2, access=3 が自然に分離、冗長ペア・横リンクの扱いは要検討)、(c) ELK 任せのまま WAN 境界だけ rank 固定。(b) は注釈不要でスケールするので有力。実測で評価すること。

### 軸C: 描画方向

- 現状は横(`direction: right`)。NW 物理図の慣習は縦(**WAN 上 → LAN 下**)。
- 縦だと冗長ペア(rt01/rt02, core01/core02)が同じ段で横並びになりやすい利点。横だと縦に積まれる。
- D2 は `direction` をコンテナ単位でも指定できる。全体と拠点内で方向を変える選択肢も検討。

### 軸D: クラウドの配置

- クラウドは**複数拠点が共有する**(IP-VPN網は全拠点から接続)ため、特定拠点コンテナ内には入れられない(概念的にも外側の共有物)。「拠点を並べて外にクラウド」という現状構造自体は正しい。
- 直すべきは **各拠点内で WAN境界機器をクラウドのある側の端に寄せる** こと。多拠点では**クラウドを全体の上部(または一方)の共有バンドに固定**する。

### 軸E: ビューの役割分担

- 「全社物理図に全 LAN 詳細を1枚」は現実の設計書でも描かない。**全社図=WAN関係+拠点は箱(`collapse_sites`)/概略、LAN詳細=拠点ごとの詳細図(`include_sites` 単一)** と役割を割り切れば、physical-all の破綻は「そもそも描かない」ことで回避できる。この方針をレンダラ/ドキュメントに反映するかを判断する。

### 未検証事項(思い込みで進めず、D2/ELK v0.7.1 で実測して確認すること)

- D2 の `grid-rows`/`grid-columns`、`near`(ピン留め)、コンテナ単位 `direction` が、**コンテナ + エッジ + ELK レイアウトと合成したときに期待通り効くか**。
- ELK の layer/rank 制約を D2 経由で与えられるか(D2 の ELK インテグレーションで何が露出しているか)。
- 縦方向にしたときのラベル(IF名・回線名)の重なり挙動。
- これらを最小のテスト .d2 を書いて確認し、確認したことを ADR に記録してから設計を確定する。

## 4. 新要件: 通信経路の可視化(正常時/障害時・プロトコル込み)

設計書では「正常時にトラフィックがどの経路を通るか」「障害時にどう迂回するか」を、可能ならその経路を決めるプロトコルまで含めて示せると価値が高い。**プロトコルレベルの経路まで出せればベスト。難しければ経路(ホップ列)と切替条件だけでも可。**

### 4.1 何を表現する必要があるか

- **経路(path)**: 始点〜終点間の順序付きホップ列(機器→機器、または IF→IF、クラウド経由含む)。例: 本社サーバ → hq-srv-sw → コア → FW → ルーター → IP-VPN網 → 大阪ルーター → 大阪サーバ。
- **正常時/障害時のシナリオ**: プライマリ経路と、特定コンポーネント障害時(例「IP-VPN 障害」「hq-rt01 障害」)に有効化されるバックアップ経路。
- **経路を決めるプロトコル/理由**: HSRP/VRRP(どのGWがactive)、BGP/OSPF(どのWAN出口か、local-pref/metric)、STP(L2 のactiveリンク)、IPsec フェイルオーバー等。各ホップまたは経路にプロトコルと選択理由を注記できると設計意図が伝わる。

### 4.2 DSL 表現の設計論点

- **明示 `paths` 方式**(推奨候補): 作成者が経路を順序付きホップ + 条件(normal / failure-of-X)+ プロトコル注記で記述する。記述負担はあるが、nwdsl は「設計意図を捉えるツール」なので明示は妥当。決定的で検証可能。
- **導出方式**(将来): 機器に冗長優先度・プロトコルメトリックを持たせて経路を計算する。ミニ・ルーティングシミュレータに近く重い。今回のスコープ外だが拡張余地として ADR に残す。
- 既存モデルとの接続: `redundancy_group`、`circuits.status`、`links`(`logical`/`tunnel` 含む)を参照して経路を組む。経路は既存 link/device を参照する形にし、二重定義を避ける。
- バリデーション: 経路のホップが実在 link/IF 上で連続していること(飛び地の経路を弾く)を validate に追加する。

### 4.3 描画の設計論点

- **経路ビュー = 既存トポロジ図へのハイライトオーバーレイ**: 経路外の要素を淡色化(dim)し、経路上のエッジを強調(太線・色・**方向矢印**)、各ホップにプロトコル/理由ラベル。
- **障害時**: 障害コンポーネントを×/赤で明示、無効化されたプライマリ経路を灰破線、有効なバックアップ経路を強調。
- D2 の機能を実測: エッジの `style.animated: true`(フロー方向の表現)、特定エッジのみの強調スタイル、方向付きエッジ(`->`)。**これらがハイライト用途で使えるか確認**してから描画方式を決める。
- ビュー種別として `type: path`(または `scenario`)を `views` に追加し、`path: <id>` や `failure: <component>` を指定する形を検討。

### 4.4 検証用サンプル

正常時/障害時が意味を持つよう、**WAN 二重化(IP-VPN プライマリ + インターネット IPsec バックアップ)+ 冗長ゲートウェイ(HSRP)** を含むサンプルを新規に用意し、「正常経路」「IP-VPN障害時のIPsec迂回」「HSRP active切替」を描けることをゴールにする。既存の `examples/sample-corp`(IPsecバックアップ+HSRPあり)を土台にできる。

## 5. 進め方の指定

1. **調査**: §2 の再現手順で問題を目視確認。§3 の未検証事項を最小テスト .d2 で実測。D2/ELK で何ができるかを事実として押さえる。
2. **設計提示**: レイアウト(問題1)と経路可視化(新要件)それぞれについて、選択肢+トレードオフ+推奨案を提示し、**承認を得てから実装に入る**。DSL スキーマを変える場合は後方互換(既存 examples が壊れないか)を必ず確認。
3. **実装**: バリデータ→ビュー解決(graph.py)→レンダラの順。1機能ごとに complex-lan と新サンプルで**実描画→PNG目視**まで確認し、崩れがあれば原因を特定してから次へ。テストを追加(現状 33 件、全緑を維持)。
4. **記録**: 設計判断は `docs/adr/` に追記(なぜその層モデル/経路表現にしたか、検討した代替と却下理由)。Phase 完了ごとにコミットを分け、メッセージに判断要約。
5. `docs/reference.md` / `docs/tutorial.md` / README を新機能に合わせて更新。コマンド例はプレースホルダを使わずコピペで動く実在パスで書き、提示前に実行検証する。
