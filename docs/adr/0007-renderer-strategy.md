# ADR-0007: レンダリングエンジン戦略 (D2の適用限界とパターン特化レンダラ計画)

- Status: Accepted (2026-07-04 改訂: 敵対的パターンの実測によりノックアウトファクタを確認)
- Date: 2026-07-04

## Context

「レンダリングをD2に任せず理想的なレンダラを自作すべきでは」という問いに対し、当初は
scale-50 (3拠点50台の木構造寄りトポロジ) の結果だけで「D2継続で十分」と判断した。
しかし「試験パターンが少なく小規模すぎる。そのまま使えるなら皆D2で構成図を出している」
という指摘を受け、**D2を落とすことを目的とした敵対的パターン4種** (examples/stress、
generate.py で再現可能) を追加実測した。

## 実測結果: ノックアウトファクタは実在する

| パターン | 判定 | 症状 |
|---|---|---|
| leaf-spine ファブリック (K4×12 完全二部グラフ) | **ノックアウト** | 48本の交差リンクが直交配線で迷路化し判読不能。layered レイアウトの構造的限界 |
| メトロリング 8拠点 (専用線で閉路) | **ノックアウト** | リングが縦一列に展開され閉路の1辺が全長を縦断。「輪」として描けない |
| 12拠点 hub&spoke WAN概要 (2クラウド+IPsec+BGP) | **重度劣化** | 拠点群が分断され、回線ラベルノードが線から遊離。設計書品質に達しない |
| LAG並列リンク+デュアルホーム | 中度劣化→**解消** | 束ね表現がなく並列線が冗長 → graph層で平行lan-cableを「×N」1本に集約する機能を実装済み |

**結論: D2+ELK が設計書品質を出せるのは「木構造に近い企業キャンパス/WAN」まで**
(この領域では scale-50=3拠点50台59接続まで検証済みで実用)。リング・ファブリック・
多拠点概要という NW 実務の頻出パターンでは破綻する。

## 洞察: 破綻パターンには「決まった描き方」がある

破綻した3パターンは、汎用グラフとしては難しくても、実務では描画慣習が固定している:

- **ファブリック**: スパイン1行・リーフ1行の2段配置 + 直線斜め辺 (座標計算は自明)
- **リング**: 円環上に等間隔配置 (三角関数のみ)
- **多拠点 hub&spoke**: クラウドを中心/上部に固定し拠点を放射・格子配置

つまり「汎用レイアウトエンジンをELK並みに作る」は非現実的でも、
**パターン特化レイアウトは個々には単純な計算**であり、自前実装の勝算が十分ある。

## Decision (改訂)

1. **木構造系 (キャンパス/WAN/経路図) は D2 を継続** — この領域は検証済みで、
   エッジ向き付け・ラベルノード化・幅自動調整・LAG束ねにより設計書品質を達成している
2. **リング/ファブリック/大規模概要が実案件で必要になった時点で、
   パターン特化の render_svg.py を実装する** — RenderGraph 中間層があるため
   シリアライザ追加のみで対応可能。実装方針:
   - views に `layout: auto | tiers | ring | radial` を追加 (auto=現行D2)
   - ELK 相当の汎用アルゴリズムは作らない。上記の固定慣習レイアウトのみ実装
   - ラベル配置は自前で全制御できるため、D2で必要だった回避策は不要になる
3. LAG 束ね (×N 集約) は graph 層で実装済み (D2/Mermaid/将来のSVG すべてに効く)

## Consequences

- 「nwdsl はどんなトポロジでも綺麗に描ける」とは主張しない。適用限界
  (木構造系まで) を README/リファレンスに明記し、破綻パターンは stress サンプルで
  再現可能にしておく
- render_svg.py 着手時は ring → tiers → radial の順 (計算が単純な順) に実装する
