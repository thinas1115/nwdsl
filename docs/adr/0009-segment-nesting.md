# ADR-0009: 論理図でのセグメント内包表示 (末端機器のネスト)

- Status: Accepted
- Date: 2026-07-06
- Phase: 機能拡張

## Context

論理構成図でセグメント(VLAN/サブネット)は、これまで単一のラベル付きボックス(例:
「本社サーバ VLAN 10 / 10.1.10.0/24」)として描き、GW機器から1本の線で繋ぐだけだった。
実際にそのセグメント上にいる端末(サーバ・PC等)を明示したいという要望があり、
「セグメントの箱の中に、実際の機器を入れ子で描く」表現を検討した。

## 検討した表現方法

1. **OSPFエリアのような色分け(`domains`)** — 既存の `domains` 機構は複数拠点にまたがる
   横断的なグルーピング向けで、厳密な親子関係(1拠点の1セグメントの中に閉じる)には
   そぐわない。却下
2. **セグメントの箱の中に端末をネスト** — D2はコンテナのネストをネイティブに扱える
   ため、拠点コンテナの中にさらにセグメントコンテナを入れ子にし、その中に端末を描く
   ことが実験で確認できた(`s_hq { seg_hq_server { n_hq_srv01 } }` のような構造)。
   複数セグメントを跨ぐL3スイッチ/ルーターは、2つの箱に同時に入ることができない
   という制約はあるが、「その機器がGWかどうか」で区別すれば解決する。採用

## Decision

**判定ルール(スキーマ変更なし)**: 機器が `role: server` で、かつ参照するセグメントが
ちょうど1つだけの場合、その機器はセグメントのGWではなく「内包される末端機器」として
扱い、セグメントの箱の中に子ノードとして描く。複数セグメントに跨る機器や
router/l3switch/l2switch等のroleを持つ機器は、これまで通り拠点レベルに置き、箱へは
GW接続の線(`Vlan10` 等のIF名ラベル付き)を引く。

この判定は既存の `interfaces[].segment` 参照だけから機械的に導出できるため、
新しいスキーマフィールドは追加していない。

**対象はD2エンジンのみ**。D2はコンテナのネストを直接サポートするため実装コストが
低い一方、内蔵SVGエンジン(svg_layout.py)は「拠点」という1階層のグルーピングしか
扱っておらず、ADR-0008の重なりゼロ不変条件を保ったまま2階層目を追加するには
別途相応の実装が要る。今回はスコープ外とし、内蔵SVGでは従来通り単一ボックスの
まま描画する。Mermaidも同様に対象外。

## 実装上の注意 (孤立機器補完との相互作用)

ADR-0005補遺2で追加した「孤立機器を実在のlan-cableで補完接続する」ロジックは、
機器単体で見て「現ビューのlayersでは誰とも繋がっていない」ものを検出して
lan-cableを補って表示する仕組みだった。末端機器(例: hq-srv01)もこの検出に
引っかかるため、素朴に実装すると「セグメントの箱に内包されている」のに
「GW機器から直接の物理配線も引かれる」という二重表現になり、ラベルが密集して
見た目が崩れた(実測で確認)。

対処として、孤立機器補完ループで「相手側 (other) がセグメント内包物になる機器か」
も判定し、該当する場合は補完エッジをスキップするようにした。内包表示が「どう
繋がっているか」を既に示しているため、直接配線の線は不要と判断した。

## Consequences

- `examples/sample-corp` の `hq-srv01`(サーバ機器)で実演。論理図で「本社サーバ」
  セグメントの箱の中にhq-srv01が描かれ、IPアドレスも見える
- 既存サンプル(hq-dc-cloud 等)の `role: server` かつ単一セグメント参照の機器は、
  スキーマ変更なしに自動的にこの表示に切り替わった(dc-srv01/02 が「DCサーバ」
  セグメントの中に入れ子で描かれるようになった)
- 複数セグメントに跨るサーバ(将来のケース)は、これまで通り拠点レベル+複数本の
  GW線で描かれる(内包されない)。安全側のフォールバック
