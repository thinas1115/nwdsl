# ADR-0005: レイアウトの階層保証 (クラウド起点BFSによるエッジ向き付け)

- Status: Accepted
- Date: 2026-07-04
- Phase: レイアウト改善 (Phase A)

## Context

初版のレンダラは `direction: right` を出力するだけで、ノードの配置を ELK に丸投げしていた。その結果、複雑な LAN 構成(examples/complex-lan、17機器多段)で以下が顕在化した。

1. WAN 境界ルーターと WAN クラウドが図の反対側に置かれ、**WAN 回線の線が LAN 機器群を横断する**(発端の指摘は「WANルーターよりLAN機器が外側に見える」)
2. 拠点コンテナのタイトルと回線の線・ラベルが衝突する
3. LAN 内部の段組み(FW→コア→ディストリ→アクセス)自体は綺麗に並ぶが、これは偶然であって保証がない

## 実測で確認した事実

最小テスト .d2 による比較実験(同一トポロジ、エッジの記述順のみ変更)で以下を確認した。

- **ELK (layered) は無向エッジ `--` でも記述順 (source→target) をランク方向として使う**。`機器 -- 雲` と書くと雲が下流(下/右)に、`雲 -- 機器` と書くと雲が上流に配置される。従来の症状は links の YAML 記述順(機器→雲)をそのまま出力していたことが根本原因
- `雲 -- 機器` に統一 + `direction: down` で、雲が最上部・WAN境界ルーターがコンテナ上端・LAN が下へ降りる理想の階層になる(交差ゼロ)
- 多拠点+共有クラウドでも雲が上部バンドに並ぶ
- 中継拠点構成(拠点—WAN—拠点—WAN—拠点、中央拠点は WAN 別に独立ルーター+間に L2 ブリッジ)でも、マルチソース BFS の距離で両 WAN ルーターが上段・L2 が下段に正しく並ぶ。逆にエッジ1本の向きを誤るとルーターが最下段に落ちることも確認(向き制御が本質である証拠)
- `grid-rows` / `near` / コンテナ単位 `direction` などの D2 機能は**不要**だった(検証対象だったが、エッジ向きだけで十分と判明したため使用せず)

## Decision

1. **graph.py のビュー解決の最終段で、全エッジを「WAN側→LAN側」に向き付けする**
   - 起点: 図中のクラウド全部(マルチソースBFS)。クラウドを含まないビュー(論理図等)では WAN 境界機器(wan-circuit link の端点機器)を起点にする
   - BFS 距離の小さい側を src、大きい側を dst に統一。同距離(コア間リンク等)は記述順を維持
2. **描画方向を `direction: down`(D2)/ `flowchart TB`(Mermaid)に変更**。NW 物理図の慣習「WAN が上・LAN が下」に一致し、冗長ペアが横並びになりやすい
3. 表示は無向 `--` のまま(矢印は描かない)。向きはレイアウトにのみ作用する

## 検討した代替案

- **全機器に明示 `tier:` を持たせて層を強制** — 却下。role→tier の単純対応は破綻する(l3switch がコアにもディストリにも使われる)ため結局手書きになるが、BFS 距離で同等の結果が注釈ゼロで得られることを実測で確認した
- **D2 の grid-rows / near による位置固定** — 却下。エッジ向きだけで足りるため複雑化に見合わない
- **ELK の layer 制約を直接指定** — 却下。D2 経由で露出しておらず、露出していても BFS 案より複雑

## 将来の拡張余地

- 中継拠点が連なる直列構成では、単一起点 BFS + `direction: right` にすると「拠点—網—拠点—網—拠点」が一直線に展開する読み方も実測で良好だった。views のオプション(例: `flow: chain`)として追加できる(未実装)

## 補遺: ラベル衝突の根絶 (同日追記)

縦レイアウト化後もラベル衝突が3種残ったため、それぞれ実測比較で対処した。

1. **拠点タイトル × 線/IFラベル**: D2 コンテナのタイトルはデフォルトで上端中央に置かれ、上から入る WAN 線と正面衝突する。`label.near` を top-left(内側)/ outside-top-center / **outside-top-left** で比較し、狭い拠点でも一切接触しない outside-top-left を採用
2. **平行 LAN 配線の中央ラベル同士**: 冗長ペア→同一SWの平行エッジでは中央ラベル(`Gi0/0/1 - Gi1/0/1` 形式)が同じ高さに並び衝突する。ELK は平行エッジの横間隔を公開していない(nodeNodeBetweenLayers 等はレイヤ間のみ)ため、**LAN 配線の IF 名は両端の機器側小ラベル (arrowhead label) に分散配置**し中央ラベルを廃止
3. **多ポート機器での端点ラベル密着**: (2) の結果、コアSWなど接続本数の多いノードでは端点ラベルが辺上で密着する。**片辺のエッジ本数 × 90px をノードの最小幅として自動設定**し、ELK が接続点を辺全体に分散することで解消(17機器・コア5本接続の complex-lan で検証)
- WAN 回線の IF 名は端点ラベルではなく中央ラベルの2行目 `(Gi0/0/0)` に置く。縦レイアウトでは WAN 線同士が空間的に分離しており衝突しない一方、端点ラベルだと1機器に複数 WAN が刺さる場合に密着することを実測で確認したため

## Consequences

- LAN がどれだけ深くなっても「上から下へ降りる」読み方が保たれる(complex-lan で検証済み)
- 全社図に全 LAN 詳細を詰め込む使い方は依然推奨しない。全社図は `collapse_sites`、LAN 詳細は `include_sites` の拠点詳細図という役割分担をドキュメントに明記する
- 接続本数の多い機器はノードが横長になる(情報量に比例した面積であり、実務の構成図の慣習とも整合)
- Mermaid 出力では端点ラベルを表現できないため LAN の IF 名は表示されない(詳細図は D2、概要・埋め込みは Mermaid という役割分担どおり)
