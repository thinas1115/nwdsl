# ADR-0002: 新規 YAML ベース DSL の策定と全体アーキテクチャ

- Status: Accepted
- Date: 2026-07-04
- Phase: 0 (調査)

## Context

[ADR-0001](0001-prior-art-survey.md) の調査により、要件を満たす既存フォーマットが存在しないことを確認した。次の判断は「どんな形式の DSL にするか」。

## Decision

### 1. 記述形式: YAML(独自文法は作らない)

検討した選択肢:

| 選択肢 | 判断 | 理由 |
|---|---|---|
| **YAML + スキーマ検証** | **採用** | AI・人間の双方が読み書きしやすい。コメント可。既存エコシステム(エディタ支援・パーサ)をそのまま使える |
| JSON | 却下 | コメント不可・末尾カンマ等で人間の編集に不向き。YAML は JSON のスーパーセットなので JSON 入力も自動的に受理できる |
| 独自文法 (D2 風のテキスト DSL) | 却下 | パーサ・エディタ支援・フォーマッタを全部自作する必要があり、得られるのは僅かな記述量削減のみ。AI パース性も専用文法より YAML が安定 |
| TOML | 却下 | 深いネスト・配列の多い構造に不向き |

### 2. アーキテクチャ: 「意味のソース」と「見せ方」の分離

課題1(用途に応じた描き分け)への回答として、DSL を2層に分ける。

```
[トポロジ定義]  sites / devices / circuits / links … 事実の単一ソース
[ビュー定義]    views: どの範囲を・どのレイヤで・どうグルーピングして見せるか
      ↓ nwdsl CLI
[検証]  参照整合性・意味的整合性
[出力]  D2 (→SVG/PNG) / Mermaid / Markdown表
```

- **トポロジ定義**が唯一の事実(Single Source of Truth)。図も表もここから導出するため、課題4(図と表の不整合)は構造的に発生しない
- **ビュー定義**は「物理レイヤだけ」「この拠点だけ」「拠点を1ノードに畳む(全社概要図)」といった描き分けの宣言。座標や色などの見た目そのものは持たず、レイアウトは出力先エンジンに委譲する
- 見た目制御を DSL に持ち込みすぎた drawthe.net の反省(座標手打ち)を踏まえ、ビュー定義は**フィルタと抽象度の宣言に限定**する

### 3. 線の意味: link に必須の `type` を持たせ、キャリア回線は独立エンティティに

課題2(線の複数の意味)への回答:

- `links[].type` を必須とし、`lan-cable`(構内物理) / `wan-circuit`(キャリア回線経由) / `logical`(論理隣接) / `tunnel`(VPN 等のオーバーレイ)を区別する
- キャリア回線契約は NetBox の Circuit に倣い `circuits` として独立させ、`wan-circuit` タイプの link から `circuit:` で参照する。「契約」(事業者・回線ID・帯域・月額)と「結線」(どの機器のどの IF に落ちるか)を分離する

### 4. 出力先: D2 をプライマリ、Mermaid をセカンダリ

- D2 v0.7.1 を本環境に導入し描画品質を実機検証済み(ADR-0001)。ELK レイアウト・コンテナ・線種制御・日本語ラベルすべて良好
- Mermaid (flowchart) は品質で劣るが GitHub / Obsidian でネイティブ描画されるため、設計書 Markdown への埋め込み用に併設する
- SVG→PNG 変換はヘッドレスブラウザ(Edge)で行う(検証済み)

## Consequences

- YAML の階層だけでは表現しづらい相互参照(link → interface)は ID 参照で表現するため、バリデータによる参照整合性検査が必須になる(Phase 2 の中核)
- ビュー定義の表現力(フィルタ・畳み込み)が本 DSL の独自価値であり、Phase 1 で最も設計検討を要する
- 出力先を2系統持つため、レンダラは「ビュー解決(グラフ変換)」と「シリアライズ(D2/Mermaid 出力)」を分離して実装する
